01 Core Goals & Foundations
Chapter 1 · 30 terms

The three properties every data system is judged by — and the vocabulary of load and performance.

Data-intensive application
An app where the volume, complexity, or rate of change of data is the main challenge — not raw CPU.
Data system
A combination of databases, caches, queues, indexes etc. composed to serve an application.
Reliability
The system keeps working correctly even when hardware, software, or humans go wrong.
Fault
A single component deviating from its spec (a disk dies, a bug fires).
Failure
The system as a whole stops providing its service to users.
Fault tolerance
Designing so that faults do not turn into failures.
Chaos Monkey
A tool that deliberately injects faults in production to test fault tolerance.
Hardware fault
A physical component failing — disk, RAM, power, network.
Software error
A systematic bug that can take down many nodes at once.
Human error
Operator or configuration mistakes — the leading cause of outages.
Scalability
The system's ability to cope with increased load.
Load parameter
A number that describes load — requests/sec, read:write ratio, fan-out.
Throughput
How many records or requests are processed per unit of time.
Response time
Time from a client sending a request to receiving the response — what the user feels.
Latency
The duration a request waits to be handled (often used loosely for response time).
Percentile (p50/p95/p99)
The value below which that percentage of response times fall.
Median (p50)
The typical response time — half are faster, half slower.
Tail latency
The slow, high-percentile responses (p99+), often hitting your most valuable users.
Tail latency amplification
When one request fans out to many backends, the single slowest one dominates.
SLA / SLO
Service Level Agreement / Objective — promised performance and availability targets.
Fan-out
The number of downstream requests or operations triggered by one input.
Vertical scaling (scale up)
Coping with load by using a more powerful single machine.
Horizontal scaling (scale out)
Coping with load by spreading across many machines (shared-nothing).
Elastic
Automatically adds computing resources as load rises.
Maintainability
How easy a system is to operate, understand, and change over time.
Operability
Making it easy for operations teams to keep the system running smoothly.
Simplicity
Managing complexity so the system stays easy to understand.
Accidental complexity
Complexity from the implementation, not inherent in the problem; remove via abstraction.
Abstraction
Hiding detail behind a clean interface so people can work without knowing the internals.
Evolvability
Making it easy to change the system as requirements evolve.
02 Data Models & Query Languages
Chapter 2 · 25 terms

The shape you give data, and the languages you use to ask questions of it.

Data model
How data is structured and the operations allowed on it — the layer that shapes everything above.
Relational model
Data as tables (relations) of rows; relationships expressed through joins.
Document model
Self-contained nested records, JSON-like trees; great for one-to-many data.
Graph model
Data as vertices (things) and edges (relationships); great for many-to-many.
NoSQL
"Not only SQL" — non-relational stores driven by scale and schema flexibility.
Impedance mismatch
The awkward translation gap between in-code objects and relational tables.
ORM
Object-Relational Mapper — library that bridges objects and tables (hides, never removes, the mismatch).
Normalization
Storing each piece of data once and referencing it by ID to avoid duplication.
Denormalization
Deliberately duplicating data to make reads faster.
Locality
Keeping related data physically together so it can be read in one go.
One/many-to-many relationship
Cardinality of links between records — the deciding factor in model choice.
Hierarchical model
1960s–70s tree model (IBM IMS); the document model's ancestor.
Network model (CODASYL)
Records linked by manually navigated access paths; hard to change.
Schema-on-write
Schema enforced when data is written (relational) — like static typing.
Schema-on-read
Structure assumed only when data is read (document) — like dynamic typing.
Declarative query
Specifies WHAT result you want (SQL); the engine decides how to get it.
Imperative query
Specifies the step-by-step HOW; harder to optimize and parallelize.
Query optimizer
The engine component that turns a declarative query into an efficient plan.
MapReduce (query)
Querying via two pure functions — map (emit pairs) then reduce (combine).
Property graph
Vertices and edges each with IDs, labels, and key-value properties (Neo4j).
Cypher
Neo4j's declarative graph query language — you draw the pattern you want.
Triple-store
Data stored as (subject, predicate, object) facts; basis of RDF.
SPARQL
The query language for RDF triple-stores.
Datalog
A foundational rules-and-facts query model; composable and powerful.
Foreign key
A field referencing the primary key of a row in another table.
03 Storage & Retrieval
Chapter 3 · 32 terms

How databases actually put bytes on disk and find them again — and how analytics storage differs.

Storage engine
The part of a database that physically stores and retrieves data.
Index
An extra structure that speeds up reads at the cost of slower writes.
Append-only log
Writing every change sequentially to the end of a file — fast and durable.
Segment
A bounded chunk of an append-only log file.
Compaction
Merging segments and discarding overwritten or deleted keys to reclaim space.
Tombstone
A marker recording that a key was deleted.
Hash index
An in-memory map of key to byte-offset in the log (Bitcask); keys must fit in RAM.
SSTable
Sorted String Table — a segment file kept sorted by key, enabling cheap merges.
LSM-tree
Log-Structured Merge-tree: memtable + SSTables + compaction; write-optimized.
Memtable
An in-memory sorted buffer holding recent writes before flushing to disk.
Bloom filter
A compact probabilistic set that quickly says "this key is definitely not here."
B-tree
Fixed-size pages in a shallow tree, updated in place; read-optimized; the relational default.
Page
A fixed-size block (often 4 KB) — the unit a B-tree reads and writes.
Branching factor
The number of child references in one B-tree page (often hundreds).
Write-ahead log (WAL)
Appending each change before applying it, so a crash mid-write is recoverable.
Write amplification
One logical write causing several physical disk writes; wears SSDs and limits throughput.
Clustered index
An index that stores the row data itself alongside the key.
Covering index
An index holding enough columns to answer a query without touching the row.
Secondary index
An index on a non-primary-key attribute (e.g. colour).
In-memory database
Keeps the whole dataset in RAM for speed (Redis, VoltDB).
OLTP
Online Transaction Processing — fast few-row reads/writes driven by app users.
OLAP
Online Analytical Processing — large aggregations for analysts and dashboards.
Data warehouse
A separate read-only analytics copy of data, loaded via ETL.
ETL
Extract, Transform, Load — the pipeline moving data into the warehouse.
Star schema
A central fact table surrounded by dimension tables.
Snowflake schema
A star schema whose dimensions are further normalized into sub-dimensions.
Fact table
One row per event (a sale, a click) — the centre of a star schema.
Dimension table
The who/what/where/when/why context referenced by the fact table.
Column-oriented storage
Storing each column together so a query reads only the columns it needs.
Column compression
Compressing repetitive column values (e.g. bitmap encoding) for fast scans.
Materialized view
A precomputed query result stored on disk for instant reads.
Data cube
Precomputed aggregates across several dimensions for fast analytics.
04 Encoding & Evolution
Chapter 4 · 20 terms

Turning data into bytes and back — and letting old and new code coexist as schemas change.

Encoding / serialization
Turning in-memory objects into a byte sequence for storage or network.
Decoding / parsing
Turning a byte sequence back into in-memory objects.
Schema / IDL
A declared structure for data; the contract between writer and reader.
Field tag
A numeric identifier for a field, written in Thrift/Protobuf bytes instead of the name.
Language-specific format
Built-in serializers (pickle, Java Serializable) — convenient but unsafe for durable data.
JSON / XML / CSV
Text formats — readable and universal but verbose and schema-optional.
MessagePack / BSON
Binary variants of JSON; modestly smaller but keep field names.
Thrift
A schema-based binary format using numeric field tags (from Facebook).
Protocol Buffers
A schema-based binary format using numeric field tags (from Google).
Avro
A schema-based binary format with no tags; matches fields by name.
Writer's / reader's schema
Avro's two schemas — what produced the bytes vs what the consumer expects.
Schema registry
A central store of schema versions used by producers and consumers.
Backward compatibility
New code can read data written by old code.
Forward compatibility
Old code can read data written by new code.
Rolling upgrade
Deploying new code to nodes gradually, so old and new run together for a while.
Dataflow
The path encoded data takes between processes (database, service, or message broker).
REST
A service style built on HTTP conventions — URLs, verbs, status codes.
RPC
Remote Procedure Call — makes a network call look like a local function call.
Message broker
A middleman (Kafka, RabbitMQ) for asynchronous, decoupled message passing.
Actor model
A concurrency model where independent actors communicate only by messages.
05 Replication
Chapter 5 · 27 terms

Keeping copies of data on several machines — and the lag and conflicts that follow.

Replication
Keeping the same data on multiple nodes for availability, latency, and read scale.
Replica
One copy of the data, on one node.
Leader / follower
Leader accepts writes and streams changes to followers, which serve reads (master/slave).
Replication log
The stream of changes a leader sends to its followers.
Statement-based replication
Shipping the literal write statements; breaks on nondeterministic functions.
WAL shipping
Replicating by sending the storage engine's write-ahead log.
Logical (row-based) log
Replicating row-level change records, decoupled from the storage engine.
Synchronous replication
Leader waits for a follower's acknowledgement — fresh, but blocks if it's down.
Asynchronous replication
Leader doesn't wait — fast, but followers lag and recent writes can be lost.
Semi-synchronous
One follower synchronous, the rest asynchronous — a common middle ground.
Failover
Promoting a follower to leader when the leader fails — full of edge cases.
Split brain
Two nodes both believe they are leader and both accept writes; data diverges.
Replication lag
How far behind the leader a follower currently is.
Eventual consistency
Replicas converge "eventually" if writes stop; meanwhile reads may be stale.
Read-your-own-writes
Guarantee that you always see your own latest writes.
Monotonic reads
Guarantee that successive reads never appear to move backwards in time.
Consistent prefix reads
Guarantee that writes are seen in causal order (no answer before its question).
Multi-leader replication
Allowing several nodes to accept writes — good for multi-region, causes conflicts.
Write conflict
Two leaders accepting different writes to the same record.
Last-write-wins (LWW)
Resolving conflicts by keeping the latest timestamp — simple but loses data.
CRDT
Conflict-free Replicated Data Type — a data type that merges concurrent edits automatically.
Leaderless replication
No leader; the client writes to and reads from several replicas (Dynamo, Cassandra).
Quorum (w + r > n)
Requiring enough overlapping replicas that reads always catch the latest write.
Read repair
Writing a fresh value back to stale replicas noticed during a read.
Anti-entropy
A background process that copies missing data between replicas.
Sloppy quorum / hinted handoff
Accepting writes on temporary stand-in nodes during outages, handed back later.
Version vector
Per-replica counters that detect whether two writes were concurrent.
06 Partitioning
Chapter 6 · 17 terms

Splitting a dataset too big for one machine — evenly, without hot spots.

Partitioning / sharding
Splitting a dataset into pieces spread across nodes, for scale.
Partition (shard)
One piece of the dataset, living on a node (usually also replicated).
Hot spot
A partition or key receiving disproportionate load — the thing to avoid.
Skew
Uneven distribution of data or load across partitions.
Key-range partitioning
Assigning contiguous key ranges per partition; good scans, hot-spot risk.
Hash partitioning
Assigning by hash of the key; even load but no efficient range scans.
Consistent hashing
A hashing scheme that moves minimal data when nodes are added or removed.
Hot key
A single very-high-load key that hashing alone can't spread; split it manually.
Local secondary index
Index partitioned by document — each partition indexes its own data; reads scatter/gather.
Global secondary index
Index partitioned by term — targeted reads but writes touch several partitions.
Scatter/gather
Querying every partition and combining the results.
Rebalancing
Moving partitions when nodes are added or removed — move as little as possible.
Fixed-number-of-partitions
Creating many more partitions than nodes so you reassign whole partitions on change.
Dynamic partitioning
Splitting a partition when it grows too big and merging when it shrinks.
Request routing
How a client finds the node holding a given key (a form of service discovery).
ZooKeeper
A coordination service often used to hold the partition-to-node map.
MPP (massively parallel processing)
Splitting one complex analytic query into stages running across all partitions.
07 Transactions
Chapter 7 · 28 terms

Grouping reads and writes so you can pretend concurrency and faults don't exist.

Transaction
A group of reads and writes that either all commit or all abort.
Commit
Making all of a transaction's changes permanent.
Abort / rollback
Undoing all of a transaction's changes as if it never ran.
ACID
Atomicity, Consistency, Isolation, Durability — the classic transaction guarantees.
Atomicity
All-or-nothing; really means abortability if anything fails.
Consistency (ACID)
Application-level invariants hold — mostly the application's responsibility.
Isolation
Concurrent transactions don't interfere; ideally appear to run one at a time.
Durability
Once committed, data survives crashes (written to disk or replicated).
BASE
The vague opposite of ACID — Basically Available, Soft state, Eventual consistency.
Single/multi-object operation
Whether a transaction touches one row or several at once.
Isolation level
How strong the isolation guarantee is — the tunable in most databases.
Read committed
The basic level: no dirty reads, no dirty writes.
Dirty read
Seeing another transaction's uncommitted data.
Dirty write
Overwriting another transaction's uncommitted data.
Snapshot isolation
Each transaction reads from a consistent snapshot frozen at its start ("repeatable read").
MVCC
Multi-Version Concurrency Control — keep row versions so readers don't block writers.
Read skew (nonrepeatable read)
Reading the same row twice in one transaction and getting different values.
Lost update
Two read-modify-write cycles overwriting each other; one update vanishes.
Compare-and-set
Writing only if the value hasn't changed since you read it; stops lost updates.
Write skew
Two transactions read overlapping data, write different rows, and break an invariant.
Phantom
A write in one transaction changing the result of another transaction's search.
Serializability
The strongest isolation — the result equals some one-at-a-time order.
Actual serial execution
Literally running transactions one at a time on a single thread.
Stored procedure
Application logic stored and executed inside the database.
Two-phase locking (2PL)
Pessimistic serializability via locks; readers and writers block each other.
Predicate / index-range lock
Locks on a search condition or key range, used to prevent phantoms.
Deadlock
Two transactions each waiting on a lock the other holds.
SSI
Serializable Snapshot Isolation — optimistic serializability; abort transactions that conflict at commit.
08 The Trouble with Distributed Systems
Chapter 8 · 20 terms

Why distributed systems are hard: unreliable networks, lying clocks, and silent pauses.

Distributed system
Multiple machines communicating only over a network.
Partial failure
Some parts working while others fail, unpredictably — the defining difficulty.
Shared-nothing
Independent nodes connected only by a network — the common cloud architecture.
Unreliable network
A network that may lose, delay, duplicate, or reorder messages.
Network partition
A link failure that splits nodes into groups that can't reach each other.
Timeout
How long you wait before declaring a node failed — always an unreliable guess.
False positive (failure)
Declaring a healthy-but-slow node dead because the timeout was too short.
Unbounded delay
No maximum on how long a message can take — why timeouts are guesses.
Synchronous vs asynchronous network
Bounded message delay (rare) vs no delay bound (the internet).
Time-of-day clock
Wall-clock time; can jump backward on NTP correction. Bad for ordering/durations.
Monotonic clock
A clock that only moves forward; correct for measuring elapsed time.
Clock skew
The difference between two machines' clocks — breaks timestamp ordering.
NTP
Network Time Protocol — synchronizes clocks, but only imperfectly.
Process pause
A node freezing (GC, VM suspend) without noticing that time has passed.
Stop-the-world GC
A garbage-collection pause that halts the whole process, sometimes for seconds.
Lease
A time-limited grant such as leadership; dangerous if a pause outlasts it.
Quorum
A majority of nodes agreeing — the basis of "truth" no single node can override.
Fencing token
An increasing number issued with each lock grant; storage rejects stale tokens.
Byzantine fault
A node that lies or behaves maliciously, not merely crashes.
System model
The assumptions you make about timing and faults when reasoning about correctness.
09 Consistency & Consensus
Chapter 9 · 21 terms

The tools for building correct systems on faulty foundations — all reducible to consensus.

Linearizability
The strongest recency guarantee — behave like a single, always-fresh copy.
Recency guarantee
A read always reflects the latest completed write.
CAP theorem
When a network partition occurs, you must choose consistency or availability.
Causal consistency
Preserves happened-before order; the strongest model that survives partitions.
Causality / happens-before
The relation defining which events must precede which.
Partial order
Some events are ordered; concurrent ones are simply unordered.
Total order
Every pair of events is ordered relative to each other.
Lamport timestamp
(counter, node-id) giving a total order consistent with causality.
Sequence number
A monotonically increasing ID that imposes an order on operations.
Total order broadcast
Delivering the same messages to all nodes in the same order; equals consensus.
State machine replication
Feeding replicas identical ordered operations so they stay identical.
Two-phase commit (2PC)
Coordinator-driven atomic commit across nodes; blocks if the coordinator crashes.
Coordinator
The node that drives a two-phase commit transaction.
In-doubt / blocking
Participants stuck holding locks, awaiting a crashed coordinator's decision.
XA transactions
A standard for distributed transactions spanning multiple systems.
Consensus
Getting several nodes to agree on one value despite failures (needs a majority).
Paxos / Raft / Zab
Fault-tolerant consensus algorithms.
Epoch (term/ballot) number
A round identifier that lets consensus algorithms detect stale leaders.
Coordination service
ZooKeeper / etcd — packages consensus into locks, leader election, config, membership.
Leader election
Using consensus to choose exactly one leader.
Membership service
Tracks which nodes are currently alive in the cluster.
10 Batch Processing
Chapter 10 · 19 terms

Crunching a large, fixed dataset into derived output — from Unix pipes to Spark.

Batch processing
Processing a large, bounded input to produce derived output; measured by throughput.
Bounded input
A fixed, finite, known dataset (as opposed to a stream).
Unix philosophy
Small composable tools, joined by pipes, operating over immutable input.
Pipe
Connecting one program's output directly to the next program's input.
Immutable input
Source data is never modified, so jobs can be re-run safely and endlessly.
MapReduce
Map then shuffle then reduce, run across a cluster with automatic retries.
Mapper
Runs once per input record and emits key-value pairs.
Shuffle
The framework sorting and partitioning so all values for a key reach one reducer.
Reducer
Runs once per key, processing that key's grouped values.
HDFS
Hadoop Distributed File System — shared-nothing disks with blocks replicated across machines.
Reduce-side join (sort-merge)
A general join performed via the shuffle; moves all the data.
Map-side join
A join done without a shuffle when data layout allows it.
Broadcast hash join
Loading a small side into memory on every mapper to avoid the shuffle.
Partitioned hash join
Joining locally because both sides share the same partitioning.
Materialization
Writing intermediate results to disk between stages (MapReduce's cost).
Dataflow engine
Spark / Tez / Flink — model a job as a DAG and keep intermediate data in memory.
DAG
Directed Acyclic Graph of operators describing a whole job.
Lineage
A record of how data was derived, used to recompute lost partitions after a fault.
Pregel
A processing model for iterative graph algorithms.
11 Stream Processing
Chapter 11 · 22 terms

Batch processing's twin for data that never stops — events, logs, and reasoning about time.

Stream processing
Processing unbounded input event-by-event, as it arrives.
Unbounded input
Never-ending data (clicks, sensors, transactions).
Event
A small, immutable record of something that happened at a point in time.
Producer / consumer
Who emits events vs who processes them (publisher / subscriber).
Topic
A named stream grouping related events.
Message broker
A middleman that buffers and routes events between producers and consumers.
Load balancing (messaging)
Each message delivered to one consumer, to share out work.
Fan-out (messaging)
Each message delivered to every consumer, so all react independently.
Partitioned log (Kafka)
A durable append-only log; messages are retained and replayable via offsets.
Offset
A consumer's position in the log, letting it replay from any point.
Change data capture (CDC)
Turning a database's change log into a stream that keeps other systems in sync.
Event sourcing
Storing the log of events as the source of truth and deriving state by replay.
Command vs event
A request to do something vs the recorded fact that it happened.
Materialized view (stream)
Current state derived from, and rebuildable from, the event log.
Event time vs processing time
When something happened vs when your system handled it — they differ.
Window
Grouping events by time: tumbling, hopping, sliding, or session windows.
Watermark
A signal that all events up to time T have probably arrived, letting a window close.
Straggler / late event
An event that arrives after the window it belongs to has closed.
Stream join
Joining streams: stream-stream (windowed), stream-table (enrich), or table-table.
Microbatching
Chopping a stream into tiny batches to reuse batch fault tolerance (Spark Streaming).
Checkpoint
A periodic snapshot of operator state for restarting after failure (Flink).
Exactly-once semantics
Making effects appear once via idempotence or atomic output+offset commit.
12 The Future of Data Systems
Chapter 12 · 17 terms

Where it's heading: integrate tools via logs, unbundle the database, stay correct, act ethically.

Derived data
Data computed from a source of truth (an index, cache, or view derived from a database).
Source of truth (system of record)
The authoritative copy of data; everything else is derived from it.
Data integration
Keeping many specialised systems consistent with one another.
Lambda architecture
Running batch and stream systems in parallel and merging their outputs at read time.
Unbundling the database
Providing storage, indexes, and views as separate tools composed via a log.
Federation
Unifying reads — one query interface over many underlying stores.
Database inside-out
Lifting a database's internal parts into the app, with the log as source of truth.
Read path / write path
Doing derived work lazily at query time vs eagerly at write time.
End-to-end argument
Some correctness properties can only be enforced at the endpoints, not lower layers.
Idempotence
An operation safe to apply repeatedly — how exactly-once is really achieved.
Timeliness
Seeing up-to-date state; violations are temporary and self-correcting.
Integrity
No corruption or loss; violations are permanent — the property that matters most.
Coordination-avoiding
Keeping integrity without linearizable coordination, for speed and availability.
Predictive analytics
Using data to make automated decisions about people (credit, jobs, policing).
Feedback loop
A model's biased outputs shaping future inputs and reinforcing the bias.
Power asymmetry
The imbalance between those who hold data and those it describes.
Data as liability
Hoarded personal data is a hazard that leaks — collecting less is often safer.
