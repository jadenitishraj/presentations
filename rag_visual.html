RAG is an evidence pipeline, not a search box.

The project's RAG folder teaches a production pattern: convert messy sources into structured retrievable evidence, retrieve the best context for a question, then generate facts and references from that context.

Why this matters

LLMs remember general patterns, but production answers need local facts: your documents, your policies, your research notes, your sources, and your current evidence.

What students should notice

The RAG design is modular. Each file owns exactly one responsibility: parser.py, chunker.py, indexer.py, pipeline.py, retriever.py, and evaluator.py.

def ingest_file(file_path: str) -> dict:
    """Takes a file, processes it, and saves it globally to the knowledge base."""
    print(f"\n=== Starting Upload Pipeline for: {file_path} ===")
    
    # Step 1: Parse & Classify (PDFs, Markdown, Transcripts, etc.)
    parsed_dict = parse_file(file_path)
    
    # Step 2: Chunk documents based on the chosen strategy
    chunk_dicts = chunk_document(parsed_dict)
    
    # Step 3: Save to Global Vector, Graph, and BM25 Stores
    index_chunks(chunk_dicts)
    
    return {"status": "success", "chunks_created": len(chunk_dicts)}

Good retrieval starts before the database.

The classifier looks at the shape of the content, then routes it to a chunking strategy. This is a production idea: do not treat every document like plain text.

The chunking problem

A chunk must be small enough to retrieve cheaply, but large enough to preserve meaning. Bad chunking separates questions from answers, headings from paragraphs, or error messages from their stack traces.

context windowmetadataoverlap
HTML

Use structure-aware parsing so navigation, tags, and content blocks do not become noise.

Markdown

Preserve headings, bullet hierarchy, and section boundaries.

Transcript

Use semantic breaks because spoken content changes topics unevenly.

Logs

Use token-based splitting so dense technical traces stay bounded.

Reports

Use hierarchical chunks so broad sections and fine details both remain available.

StrategyBest forRisk it avoids

Code Breakdown: Notice how the chunker avoids hardcoding a single splitting rule. By reading the classification tag from the parser, it dynamically selects between Semantic chunking (for transcripts), Hierarchical chunking (for long reports), or basic fallback strategies.

backend/rag_v2/chunker.py (Dynamic Splitting Strategies)
def chunk_document(parsed_doc: dict) -> list[dict]:
    """Split a parsed document into chunks based on its classified strategy."""
    strategy = parsed_doc["chunk_strategy"]
    
    if strategy == "markdown":
        splitter = MarkdownNodeParser()
    elif strategy == "html":
        splitter = HTMLNodeParser()
    elif strategy == "token":
        splitter = TokenTextSplitter(chunk_size=512, chunk_overlap=50)
    elif strategy == "semantic":
        # Semantic chunking splits when embedding similarity drops
        splitter = SemanticSplitterNodeParser(breakpoint_percentile_threshold=90)
    else:
        splitter = SentenceSplitter(chunk_size=768, chunk_overlap=80)
        
    doc = Document(text=parsed_doc["text"], metadata=parsed_doc["metadata"])
    return [{"id": n.node_id, "text": n.text, "metadata": n.metadata} for n in splitter.get_nodes_from_documents([doc])]

Tri-index storage captures meaning, relationships, and keywords.

The project stores chunks in Chroma for vector search, uses a graph backend for relationship search, and builds a BM25 index for exact keyword matching (with a fallback graph store if Neo4j is offline).

Vector database

Embeddings turn text into coordinates. Nearby vectors usually mean nearby meaning, so a question about "repairing an engine" can find text about "fixing a motor."

Graph database

Graphs store entities and relationships. They help answer multi-hop questions where the connection matters more than word similarity.

BM25 Keyword Index

While Vectors look at meaning, BM25 looks at exact keyword matches using TF-IDF. It acts as a safety net: if a user searches for an exact product ID like "ERR-409", the vector model might misunderstand the meaning, but BM25 will find the exact string immediately.

keyword matchtf-idfexact terms

Production robustness

The storage layer is built to degrade gracefully. If the graph database is configured, the system uses it. If not, it falls back to a simple local graph store so the learning demo still works.

Code Breakdown: The indexer iterates over the chunks just once, but maps them into three distinct mathematical representations: a Vector Index for meaning, a Knowledge Graph for connections, and a BM25 store (handled dynamically) for exact string matching. Notice how BM25 isn't saved to a database—it's built in-memory on the fly during retrieval using the `processed_nodes` returned here.

def index_chunks(chunks: list[dict]):

    storage = get_global_storage()

Retrieval improves the question before searching.

Users ask short, vague questions. The system uses HyDE, vector search, BM25 keyword search, graph lookup, reciprocal reranking, and LLM reranking to narrow many candidates into the best few contexts.

Q

User query

The raw question may be ambiguous, incomplete, or use different vocabulary.

H

HyDE

Generate a hypothetical answer-shaped document to search by meaning.

V

Vector

Find semantically similar chunks even when words differ.

B

BM25

Find exact names, identifiers, and keyword matches.

G

Graph

Find relationship-based evidence when entities connect across hops.

F

Fusion

Merge candidates so no single retrieval method dominates unfairly.

R

Rerank

Select only the top context chunks for the answer writer.

BM25

Great for exact terms such as product names, error codes, file names, law names, and IDs.

Vector search

Great for meaning, paraphrases, concept matching, and questions that use different wording from the source.

LLM rerank

Great for final selection because it compares the question against candidate chunks directly.

Code Breakdown: The retriever is the heaviest lifter. It first expands the query using HyDE, then hits the Vector and BM25 indexes simultaneously. It merges those candidates using Reciprocal Rank Fusion, adds Graph hits, and finally runs a cross-encoder LLM to pick only the top 3 best chunks.

backend/rag_v2/retriever.py (Hybrid Fusion & Rerank)
def search_global_db(query: str, top_k: int = 3) -> dict:
    # 1. Setup keyword (BM25) and semantic (Vector) search
    print("  → Searching Vector and BM25 indexes...")
    vector_retriever = vector_index.as_retriever(similarity_top_k=top_k * 2)
    bm25_retriever = BM25Retriever.from_defaults(nodes=nodes, similarity_top_k=top_k * 2)
    
    # 2. Fuse Vector + BM25 using Reciprocal Rank Fusion
    print("  → Fusing Vector and BM25 results...")
    fusion = QueryFusionRetriever([vector_retriever, bm25_retriever], mode="reciprocal_rerank")
    candidates = fusion.retrieve(QueryBundle(query_str=query))
    
    # 3. LLM Reranking of final candidates
    print("  → LLM reranking top candidates to find the best chunks...")
    reranker = LLMRerank(llm=get_llama_llm(), choice_batch_size=5, top_n=top_k)
    best_nodes = reranker.postprocess_nodes(candidates, query_bundle)
    
    return {"query": query, "results": [{"text": n.node.text} for n in best_nodes]}

Citations complete the trust loop

Retrieval is not only about feeding the model. It also gives users and evaluators a way to inspect where the answer came from. Citations make the answer auditable.

Ragas-style evaluation

After retrieval and answer generation, the system can score whether the answer is faithful to the retrieved context, whether the context was relevant, and whether the answer is supported by evidence. This turns RAG quality into a measurable report instead of a feeling.

Code Breakdown: The evaluator uses Ragas to act as an impartial judge. It spins up 4 independent LLM evaluations to test the answer against the retrieved context. If Faithfulness drops, the LLM hallucinated. If Context Precision drops, the retriever surfaced bad chunks.

backend/rag_v2/ragas_evaluation.py (Ragas Metrics)
def evaluate_ragas(question: str, answer: str, contexts: list[str]) -> dict:
    print("\n=== Starting Ragas Evaluation ===")
    dataset = Dataset.from_dict({
        "question": [question],
        "answer": [answer],
        "contexts": [contexts],
    })
    
    # LLM-as-a-Judge computes these 4 independent scores
    metrics = [
        faithfulness,
        answer_relevancy,
        context_precision,
        context_recall
    ]
    
    return evaluate(dataset, metrics=metrics, llm=get_langchain_llm())
05

Vocabulary and final recap.

These terms let students explain the RAG system like engineers instead of treating it as magic.

RAG

Retrieval-Augmented Generation: answering with retrieved evidence instead of only model memory.

Chunk

A retrievable piece of a document, sized to preserve meaning and fit context limits.

Metadata

Extra information such as title, URL, category, and chunking strategy attached to each chunk.

Embedding

A numerical representation of text meaning used for similarity search.

Vector DB

A database optimized for finding nearby embeddings.

Graph DB

A database optimized for entities, relationships, and multi-hop reasoning.

BM25

A keyword ranking method that rewards exact term matches and useful word frequency.

HyDE

A query transform that creates a hypothetical answer document to improve semantic retrieval.

Reranking

Scoring candidate chunks again so only the most relevant context reaches the final answer.

Ragas

A family of RAG evaluation ideas for scoring faithfulness, context relevance, and answer quality.

Final summary

  • RAG turns messy knowledge into searchable evidence for an LLM.
  • Classification matters because HTML, Markdown, transcripts, logs, and reports need different chunking strategies.
  • Vector search finds meaning; BM25 finds exact terms; graph retrieval finds relationships.
  • HyDE and query transformation help when the user's question is vague or uses different language from the source.
  • Reranking, citations, and Ragas-style scorecards make the final answer more relevant, cheaper, and easier to audit.
