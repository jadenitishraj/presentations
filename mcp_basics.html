01

A model can only talk. It can't do.

By itself, a language model is just a very good text writer. Ask it "what's the weather right now?" or "read my Notion page" and it is stuck. It has no hands. It can't reach out into the world.

A raw model is a brilliant brain locked in a room with no doors.

It can reason, write, and explain. But it cannot look up today's news, open a file on your computer, search the web, or send a message. All of those need an action in the real world — and a model only produces words.

So we give it tools. A tool is just a small action the model is allowed to ask for: "search this," "read that file," "send this email." The model says which tool it wants and what to pass in. Something else runs it and hands back the answer.

The core idea

The model is a smart manager who can't leave the desk. Tools are the staff who go out, do the task, and bring back the result. The manager decides; the staff act.

02

Start simple: a local tool.

The easiest tool is just a Python function you write yourself. In LangGraph, you stick @tool on top of a function and it becomes something the model can call. Nothing fancy — it lives in your own code, on your own machine.

Let's say we're building a research agent. It has four helpers working together: a planner (decides the steps), a searcher (finds info), a writer (drafts the answer), and a critic (checks the work).

The searcher needs two tools to do its job: a way to search the web (DuckDuckGo) and a way to look inside our own documents (a RAG retriever). Both start life as plain local functions.

tools.pyLangGraph · @tool
from langchain_core.tools import tool @tool def search_web(query: str) -> str: """Search the web for a query and return the top results.""" return duckduckgo(query) # your own code @tool def retrieve_docs(query: str) -> str: """Find passages from our own documents (RAG).""" return vector_store.similarity_search(query)
THE DOCSTRING IS NOT DECORATIONThat line in triple-quotes is the tool's description. The model reads it to decide when to use the tool. A vague docstring = the model never picks it. Write it like you're explaining to a new intern.
WHERE IT LIVESThis function runs inside your own program, on your own machine. That's why we call it a local tool. No network, no server, no setup. Perfect to start with.
03

The model only knows the tools you hand it.

This trips everyone up, so say it plainly: the model does not magically discover your tools. You collect them into a list and pass that list into the model. It can only pick from what's in front of it.

So the job splits in two. You gather the tools and give them to the model once. The model, every turn, reads the names and descriptions, picks the right tool, and fills in the arguments (like {"query": "AI news"}). You never hardcode which tool gets called — the model decides.

04

Now the real problem. How do I share my tool?

My RAG retriever is great. I want my whole team — and their agents — to use it too. But a local function only works inside my code. The moment someone else wants it, the pain begins.

Think about what it takes to use any outside service today. Every company's API is different. Different login. Different web address. And worst of all — every one sends back a different shape of JSON.

Notion sends
{
  "results": [
    {"properties":
      {"title": [...] }}
  ]
}
Gmail sends
{
  "messages": [
    {"id": "...",
     "payload":
       {"headers":[...]}}
  ]
}
Slack sends
{
  "ok": true,
  "messages": [
    {"text": "...",
     "ts": "..."}
  ]
}

So for every service you must: read their docs, write login code, call the right address, then dig the useful text out of their odd JSON and re-shape it for the model. Then do it all over again for the next service. And the next. This glue code is boring, fragile, and never ends.

The frustration

It's like every wall socket in the world having a different shape. Every device needs its own custom plug, hand-made. You'd spend all your time making plugs and none of your time using the device.

05

MCP: one standard plug for every tool.

MCP stands for Model Context Protocol. "Context" = the data and tools a model needs but doesn't carry inside itself. "Protocol" = one agreed set of rules everyone follows. Put together: a standard way to hand a model tools it doesn't already have.

The fix is simple and powerful. Instead of you writing glue for each service, the service author writes it once and wraps it in an MCP server. Now every server — Notion, Gmail, yours — speaks the same language. You plug in and get tools in one fixed shape. No more custom plugs.

Without MCP

write glue for every service
  • Read every API's docs
  • Write separate login for each
  • Decode each one's messy JSON
  • Re-shape text for the model
  • Repeat forever, for every tool
→

With MCP

plug in, get tools
  • Author wrote the wrapper once
  • One standard way to connect
  • Tools arrive in one fixed shape
  • Model already understands it
  • You write zero glue
The famous line

MCP is the USB-C for AI. One plug shape. Any device, any model, any tool — they all fit the same port.

06

The one shape every reply follows.

Here's the heart of "standard." No matter which server you call — calculator, Notion, Gmail — the answer always comes back wrapped the same way: a content list, holding text. The model knows how to read this every time.

any tool replysame for ALL servers
"result": { "content": [ { "type": "text", "text": "Here is the answer..." } ], "isError": false }
content

Always a list. The actual result lives inside it.

type

Usually "text" — what the model needs. Can also be image, audio.

isError

Simple flag. true if the tool failed. Otherwise the result is good.

WHY THIS IS THE WHOLE TRICKThe server author decodes the messy JSON once and packs the useful text into this box. So you — and a thousand other people — never have to. Decode once, reuse everywhere. That is the entire value of MCP in one picture.
07

Build your own server — it's tiny.

Remember our problem: we wanted to share the RAG retriever. The answer is to wrap it in an MCP server. With FastMCP (Python), this is almost no work. You write normal functions; the library turns them into MCP tools.

calc_server.pyFastMCP · the entire file
from fastmcp import FastMCP mcp = FastMCP("Calculator") @mcp.tool() def add(a: float, b: float) -> float: """Add two numbers.""" return a + b @mcp.tool() def divide(a: float, b: float) -> float: """Divide a by b.""" if b == 0: return "Error: cannot divide by zero" return a / b if __name__ == "__main__": mcp.run()
@mcp.tool()

One decorator turns a plain function into an MCP tool. That's the whole magic.

the docstring

Becomes the tool's description — what the model reads to decide. Write it well.

just return

You return a value. FastMCP wraps it into the content → text shape for you, automatically.

RUN ITpip install fastmcp  then  python calc_server.py. Your retriever becomes a server the exact same way — wrap the function, done. Now anyone can plug into it.
08

The client is just a waiter.

The server holds the tools. The model decides what it wants. The client sits in the middle and carries messages back and forth. It decides nothing on its own — it just connects, fetches the tool list, and runs whatever the model chose.

1

Connect

The client opens a line to the server.

2

List tools

It asks "what can you do?" and gets the tool list back. client.list_tools()

3

Hand tools to the model

You pass that list into the LLM so it can choose.

4

Carry the call

When the model picks a tool, the client runs it on the server and brings the answer back.

Remember

Server = the kitchen (has the food). Client = the waiter (carries orders). Model = the customer (decides the order). The waiter never decides what you eat.

09

One client. Many servers.

A single client can connect to many servers at once. It collects all their tools into one big pool and hands the whole pool to the model. The model picks any tool from any server; the client quietly routes the call to the right one.

client.pyFastMCP · one client, many servers
from fastmcp import Client client = Client({ "calculator": {"command": "python", "args": ["calc_server.py"]}, "notion": {"url": "https://mcp.notion.com/mcp", "auth": "oauth"}, "gmail": {"url": "https://gmailmcp.googleapis.com/mcp/v1", "auth": "oauth"}, "playwright": {"command": "npx", "args": ["-y", "@playwright/mcp"]}, }) async with client: tools = await client.list_tools() # all tools, one list
NAMES GET A PREFIXTo stop clashes, tools come back named by server: calculator_add, notion_search, gmail_search_emails. The model reads each name + description and picks.
WHATEVER THE SERVER PAGE GIVES YOUWhen you find a server online, it shows a config block. You just drop that block inside the client. That's the whole setup.
10

Two kinds of server: local and remote.

Some servers run on your machine (because they touch your machine). Some run in the cloud on the company's machine. The first you start yourself; the second you just connect to.

Local server · stdio

runs on your machine
  • Touches local things: your files, your browser
  • Examples: Playwright, Filesystem, your calculator
  • The client starts it for you
  • Config uses command + args
  • Talks over stdio (the default)
vs

Remote server · HTTP

runs in the cloud
  • Lives on the company's servers
  • Examples: Notion, Gmail, GitHub
  • Already running — you just connect
  • Config uses a url
  • Talks over HTTP, usually needs login

Decoding the config: {"command": "python", "args": ["calc_server.py"]} simply means "start this server by running python calc_server.py" — the same line you'd type in your terminal. python is the program; calc_server.py is the file it runs. The client launches it as a background process and talks to it over stdio.

PLAYWRIGHT = THE PERFECT EXAMPLEPlaywright drives your browser — opens pages, clicks buttons. That can only happen on your machine, so it must be local. Cloud servers like Notion can't touch your browser, so they live remotely.
11

Logging in to Notion & Gmail.

Your calculator needs no login. But Notion and Gmail hold private data, so they ask "who are you?" first. This is OAuth — and the nice part is you never paste a secret token by hand.

In code it's one word: auth="oauth". The first time, FastMCP opens your browser, you log in and approve, and it saves the token to a file (~/.fastmcp/oauth-mcp-client-cache). After that, no login — it reuses the saved token.

On your laptop · testing

The browser popup is fine. You're sitting there to click "approve." Easy.

Deployed project · no browser

A cloud server has no screen to pop a browser. So you get the token once, store it as an env variable / secret, and the code reads it. No popup.

OAUTH vs API KEYNotion & Gmail use OAuth (browser login). Some smaller servers use a plain API key instead — then you just pass the key: auth="your-key". Either way, you don't hand-write login code.
12

The payoff: club every tool together.

This is where it all comes home. Your local @tool functions, your own MCP server, and outside servers like Notion and Gmail — they all become one kind of thing, and you pour them into a single list for the model.

agent.pylocal + MCP, all in one list
from langchain_mcp_adapters.client import MultiServerMCPClient mcp_client = MultiServerMCPClient({ # ...calc, notion, gmail... }) mcp_tools = await mcp_client.get_tools() # MCP → LangChain tools all_tools = [search_web, retrieve_docs, *mcp_tools] # club them llm_with_tools = llm.bind_tools(all_tools) # hand to model
THE KEY INSIGHTAfter the adapter runs, a local @tool and an MCP tool are the same type — both are LangChain tools. That's why you can mix them in one list freely. The model can't even tell which came from where.
① local @tool ② your MCP server ③ outside servers → one list → LLM
13

An honest correction: MCP does not beat timeouts.

A common myth says "use MCP instead of APIs because APIs time out." Not true. MCP times out too — around 30–60 seconds. If your tool runs for 20 minutes, plain MCP fails just like an API would.

The real fix for long jobs isn't MCP — it's working async: the tool starts the job, hands back a job ID right away, and the agent checks back later for the result. MCP recently added this as a feature called Tasks (still experimental).

The myth

wrong reason to use MCP
  • "MCP avoids the 30s API timeout"
  • "Long tools just work on MCP"
  • Reality: MCP times out the same way
✕

The truth

right reason to use MCP
  • MCP's value = one standard plug
  • Long jobs = solved by async / Tasks
  • Start job → return ID → poll later
SO WHY USE MCP?Not for speed or timeouts. Use it because you stop rewriting glue for every service. One standard, plug in any tool. That — and only that — is the real win.
