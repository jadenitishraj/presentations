Transports connect clients to tool execution.

MCP defines how clients communicate with servers. Our system teaches two environments: in-process (in-memory) for simple operations, and standalone network tools for external platforms.

In-Process (RAM)

The client imports the Python server class directly into the running memory. There are no ports, no subprocesses, and no network cards used. It's fast and operates inside the main server's thread pool.

fastmcp clientin-memoryzero config

Stdio Subprocess

The client spawns the server as a child process (e.g. Node or Python CLI) and sends JSON-RPC messages via standard input/output. This keeps environments isolated.

stdio pipechild processisolated env

SSE (HTTP/Web)

Server-Sent Events allow the server to expose tools over HTTP. An external client (like Claude Desktop or another machine) calls tools by connecting to the HTTP SSE endpoint.

http / ssefastapi mountnetwork access

Why In-Process makes testing simple

For students, setting up multiple terminal sessions for servers can be confusing. FastMCP allows us to import the server directly and run it in-process. The backend treats it as a protocol-compliant MCP server, but it executes locally in RAM.

Command-Line Standalone Testing

We can launch the Math server as a standalone stdio listener or run the FastMCP inspector dashboard to interact with tools visually using the CLI.

Multi-Server Host Connectors

LLM platforms (like Claude Desktop, Gemini, or Cursor IDE) act as MCP Hosts. A single host can connect to multiple independent MCP servers simultaneously, merging their tool registries dynamically so the LLM has access to all tools together.

from backend.mcp.server import mcp as math_server
from fastmcp import Client

async def _call_tool(name: str, args: dict) -> str:

    async with Client(math_server) as client:
        result = await client.call_tool(name, args)
        return result.content[0].text

Overcoming Notion search API limits.

Notion's search API only matches page/database titles, not page contents. We implemented a smart fallback block scanner to recursively extract nested child content and toggle blocks.

The Search Problem

If a student has a page called "Quotes" and inside that page they have sub-headers or toggle blocks like "Jiddu Krishnamurthy", searching the Notion API for "Krishnamurthy" returns zero pages because the *page title* is "Quotes", not the name of the author.

The Fallback Solution

If direct search yields no matching pages, the tool fetches **all** recent workspace pages. It reads the block content of those pages, searches block text for keywords, and if it finds a matching block with children (like a Toggle block), it recursively fetches child blocks to retrieve the nested quotes.

backend/mcp/notion_client.py (Recursive Block Fetcher)
def search_notion(query: str = "") -> str:
    # 1. Try direct title search
    results = notion.search(query=query).get("results", [])
    if not results:
        # 2. Fallback: Scan blocks in the top page (e.g. "Quotes")
        all_pages = notion.search(query="").get("results", [])
        for page in all_pages:
            blocks = notion.blocks.children.list(block_id=page["id"]).get("results", [])
            for block in blocks:
                if query.lower() in get_block_text(block).lower():
                    # 3. Read child blocks recursively (e.g., inside toggle blocks)
                    if block.get("has_children"):
                        child_blocks = notion.blocks.children.list(block_id=block["id"]).get("results", [])
                        return extract_text_from_blocks(child_blocks)

How Team Selection routes nodes

In team_selector.py, we analyze keywords. If a question is basic math (e.g., "what is 42 * 9") or references Notion, it returns a 2-agent team. Because LangGraph builds the graph shape dynamically based on this team list, the other nodes are never even compiled into the graph.

Short-circuiting the Critic

Our Critic agent has strict requirements (requiring 3+ sources, 100+ words, academic tone). If applied to a simple math question or listing Notion pages, the Critic would reject the answer. We short-circuited the Critic to immediately return `approved=True` for these special routes.

Five pillars of MCP security.

Giving LLMs access to code execution and filesystems introduces massive risks. Our codebase demonstrates several active techniques to sandbox, restrict, and guard the system.

1. Least Privilege Tool Binding

We only expose tools that are required for the current query. A general search agent never receives the API tokens or execution bindings for write/delete tools or mathematical MCP libraries.

2. Secure Secrets Handling

Keys like the NOTION_TOKEN are kept strictly in local environment configurations. They are passed directly to child server environments on startup and never exposed to git, the model, or in system logs.

3. Input & Output Guardrails

Using Microsoft Presidio, we analyze text entering and exiting the backend. This helps identify and sanitize PII (Personally Identifiable Information) before it gets forwarded to external LLM providers or tools.

4. Restricted Execution (No Shells)

Our MCP server runs isolated code blocks. It does not contain tools that accept raw shell strings or execute CLI commands. This prevents Remote Code Execution (RCE) via LLM prompt injections.

5. Production Sandboxing (When to Containerize)

The recommended architecture for production is **Dockerization**. You should containerize your MCP servers when:

  • Runtime Conflicts exist: One server uses Node.js (Notion), another uses Python (Math), and another requires Go.
  • Host protection is required: The LLM has tools that read, write, or delete files, or run command line shells. Sandboxing prevents malicious mutations on the host filesystem.
  • API Key Isolation is needed: Each container gets injected with only its specific credentials, preventing servers from reading each other's secrets.

Vocabulary and final recap.

Use these terms to explain the Model Context Protocol ecosystem clearly to your students.

MCP

Model Context Protocol: An open standard protocol for linking LLMs to tools and datasources.

Client

The host application (like LangGraph, Claude Desktop, or our FastAPI server) that queries tools and coordinates LLM calls.

Server

A standalone utility (like our Math server or Notion server) that exposes tools, resources, and prompts over a transport.

Stdio Transport

An MCP transport communicating via standard input/output streams between parent and child processes.

SSE Transport

Server-Sent Events: communicating asynchronously over HTTP, allowing remote network access.

In-Process

Running the server directly inside the client's python process memory without sockets or child processes.

FastMCP

A high-level framework from Anthropic to quickly build Python-based MCP servers and clients.

Guardrails

Verification filters (like PII scans and prompt injection filters) that intercept data before/after LLM execution.

Sandboxing

Restricting tool execution inside isolated virtual spaces (like Docker) to secure the host machine.

Final recap for students

  • MCP standardizes how LLM clients discover and call tools, removing custom API wrappers.
  • In-process transport executes tools instantly in memory, simplifying local code orchestration.
  • Notion integrations require block-level fallback scans because standard Search APIs are limited to page titles.
  • Routing simple math/Notion queries directly to Writer/Critic saves CPU cycles, time, and LLM token costs.
  • Security must be built at all layers: least privilege routing, input sanitization, and containerized sandboxing (Docker).
  • A single MCP host (Claude, Gemini, etc.) can connect to multiple MCP servers, dynamically merging their tools.
