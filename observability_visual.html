01

The problem.

A student asks one question. Under the hood, many things happen: planning, searching, reading, writing, critiquing, API calls, request logging, retries, timing, and cost. If something goes wrong, where do you look?

Without observability

The system feels magical.

You only see the final answer. If it is wrong, slow, or expensive, you are left guessing.

What students usually try first

Just print everything.

That works for tiny scripts. It breaks down once many components are active at the same time.

What observability gives you

A way to inspect reality.

Not one giant mess of text. Different views for workflow, LLM spending, and system health.

03

LangSmith.
See the workflow.

In this project, LangSmith is attached to the agents and tools. It is the place where students can inspect the life of a run: planner, searcher, reader, writer, critic, and the loops between them.

What it is trying to solve

Agent systems are hard to debug.

If the final answer is weak, the real question is not just “what was the answer?” It is “which step failed?” Did the planner create weak queries? Did search return poor sources? Did the writer ignore facts? LangSmith gives you that step-by-step picture.

How it is used here

Traceable agents and tools.

Each agent function is decorated with @traceable, and the search tool is also traced. That means each run can be opened like a tree instead of a blob of logs.

Students should notice

Runs

A run is one execution of something. One whole research request is a big run. Inside it are smaller runs for agents and tools.

Students should notice

Trace

A trace is the full journey of one request. Think of it like a timeline plus a family tree: what happened, and which step created which child step.

Students should notice

Metadata

Metadata is extra context attached to a run, like the original question or selected team. It helps you filter and understand runs later.

Vocabulary, in plain English
trace

Meaning: a step-by-step record of a request moving through the system.

Why it is used: because a final answer alone hides the path that produced it.

run

Meaning: one execution of a task, function, agent, or tool.

Why it is used: because each part of the system needs to be inspectable on its own.

tool call

Meaning: when an agent uses an outside capability, like web search.

Why it is used: because agents often fail due to bad tool usage, not just bad writing.

evaluation

Meaning: checking whether the system behaved well.

Why it is used: because “it ran” is not the same as “it worked correctly.”

04

Helicone.
See the LLM traffic.

LangSmith tells you what the workflow did. Helicone tells you what happened specifically at the LLM API layer: cost, speed, cache behavior, and request-level analytics.

What it is trying to solve

LLM calls are expensive and easy to ignore.

As soon as you have multiple agents, you no longer have one model call. You may have many. Some are fast, some are slow, some are repeated, and all of them cost money. Helicone makes those calls visible.

How it is used here

A transparent proxy.

The project routes ChatOpenAI requests through the Helicone base URL. That means the app still behaves like an OpenAI app, but Helicone sits in the middle and records what passes through.

Core idea

Proxy

A proxy is a middle layer. Instead of your app talking straight to OpenAI, it talks to Helicone first, and Helicone forwards the request onward.

Core idea

Caching

A cache stores a previous result so the same request can be served faster later. It can reduce both latency and cost when prompts repeat exactly.

Core idea

Per-agent properties

Custom request headers let this project label calls as Planner, Writer, Critic, and so on. That makes cost analysis much more meaningful for students.

Vocabulary, in plain English
proxy

Meaning: a middle service that receives your request before the final destination does.

Why it is used: because it can measure, log, and improve requests without forcing a full rewrite of your app.

request headers

Meaning: little pieces of extra information sent along with a request.

Why it is used: to attach labels like which agent made the call, or whether caching should be enabled.

latency

Meaning: how long a request takes from start to finish.

Why it is used: because a good answer that takes too long still feels broken to users.

cache hit

Meaning: the system found a saved answer instead of recomputing it.

Why it is used: because repeated prompts should not always cost full price.

05

Grafana.
See the system health.

In this project, Grafana is the wider operations view. It is not primarily about the inner thinking of agents. It is about logs, metrics, API timing, and the health of the service around the agents.

What it is trying to solve

Production systems fail in boring ways.

Sometimes the model is fine. The real issue is that the API got slow, requests piled up, errors increased, or a payload caused a failure. Grafana helps you notice those operational problems.

How it is used here

Loki for logs, Prometheus for metrics.

The project logs incoming and outgoing API traffic to Loki, and pushes metrics through OpenTelemetry into Prometheus-compatible storage for dashboards and trend watching.

The easiest distinction

Log

A log is a text record of something that happened, like an incoming request or an error message.

The easiest distinction

Metric

A metric is a number measured over time, like request count, latency, or error rate.

The easiest distinction

Dashboard

A dashboard is the visual place where those numbers and logs are assembled into something humans can read quickly.

Vocabulary, in plain English
Grafana

Meaning: the place where you view and explore observability data.

Why it is used: because raw metrics and raw logs are hard to understand without dashboards and queries.

Loki

Meaning: the log system used with Grafana.

Why it is used: to store and search text events like incoming requests, outgoing responses, and errors.

Prometheus

Meaning: the metrics system used for time-series numbers.

Why it is used: because system health is often easier to understand as changing numbers over time than as text paragraphs.

OpenTelemetry

Meaning: a common instrumentation standard for collecting observability data.

Why it is used: so apps can emit metrics and telemetry in a standard way instead of inventing everything from scratch.

middleware

Meaning: code that sits in the path between request and response.

Why it is used: because it is a natural place to measure duration and log payloads for every API call.

counter

Meaning: a metric that only goes up, like total LLM calls or total requests.

Why it is used: because totals are useful for traffic, usage, and throughput trends.

Short version If LangSmith tells you how the research pipeline behaved, Grafana tells you how the service itself behaved.
06

Which tool answers which question?

Students often confuse these tools because all of them feel like “monitoring.” The cleanest way to separate them is by the question you are asking.

Question Best Tool Why
Which agent step failed? LangSmith It shows the graph-like execution trace of agents and tools.
Why is the app getting expensive? Helicone It sits on the LLM request path and measures cost per request and per agent.
Why is the API getting slow? Grafana It tracks request timing, logs, and operational metrics.
Did the writer or critic call the model more than expected? Helicone Per-agent labeling makes those costs and patterns visible.
What exact request/response payload moved through the API? Grafana + Loki Loki is built for searching log events and payload text.
What happened across the full research run? LangSmith It keeps the run structure connected from parent to child steps.
07

How this project wires them together.

These tools are not abstract ideas here. They are attached to specific places in the codebase, each at a different layer.

Implementation map

Three layers, three attachment points.

LangSmith is attached where agents and tools run. Helicone is attached where LLM calls are made. Grafana is attached where the API receives traffic and emits metrics.

That separation is useful for teaching because students can see that “observability” is not one giant feature. It is several carefully chosen observation points.
